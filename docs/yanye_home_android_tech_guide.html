推荐总方案

妍叶之庭适合采用现代 Android 原生开发：Kotlin + Jetpack Compose + MVVM/Repository + Room + DataStore + WorkManager，后期再接 Firebase 或 Supabase 做双人同步。

首版不要追求一次做完整云端系统。先把本地闭环做顺：纪念日、愿望、日程、吃什么、备忘、姨妈记录和首页聚合。

语言

Kotlin

Android 原生新项目优先选 Kotlin。它和协程、Compose、Room、DataStore 等现代库配合最好，代码也比 Java 更简洁。

界面

Jetpack Compose

用代码直接写 UI，不再主要依赖 XML。妍叶之庭的首页卡片、状态灯、转盘、冷静模式都很适合状态驱动 UI。

数据

Room + DataStore

Room 存业务对象，DataStore 存轻量设置。这样既能承接复杂关系数据，也不会把 App 偏好混进数据库。

技术栈选择

以下组合足够支撑从 V0.1 本地版到 V1.0 双人私密版，不需要一开始引入太多重型后端能力。

选择标准：官方推荐、生态稳定、适合新手学习、后续可扩展。

领域 推荐选择 用途 为什么适合妍叶之庭
开发语言 Kotlin 写业务逻辑、数据库、页面状态 现代 Android 主流，代码短，适合长期维护。
UI 框架 Jetpack Compose + Material 3 首页、列表、表单、弹窗、导航、动效 非常适合今日面板、状态灯、冷静模式这种随数据变化的界面。
页面导航 Navigation Compose 页面跳转与底部导航 能清晰管理首页、愿望、日程、地图、我的等模块。
本地数据库 Room 纪念日、愿望、日程、回忆、餐厅、冲突复盘 你的数据对象很多，必须用结构化数据库，而不是简单文件。
设置存储 DataStore 下班提醒时间、主题、隐私开关、首次引导状态 适合保存小而稳定的用户偏好。
后台任务 WorkManager + Notification 纪念日提醒、承诺到期、姨妈期照顾提示 能处理不打开 App 时也需要提醒的场景。
依赖注入 Hilt 管理 Repository、Database、UseCase 项目变大后可以减少手动传对象的混乱。
图片加载 Coil 回忆照片、餐厅照片、挑战完成照片 Compose 生态友好，加载本地和网络图片都方便。
后期云同步 Firebase 或 Supabase 账号、双人同步、照片云存储、推送 V0.2 后再接入，避免首版被后端复杂度拖住。

App 架构

建议使用 MVVM + Repository。页面只负责展示，ViewModel 负责状态，Repository 负责数据来源，Room 和云端同步都藏在数据层里面。

这样后期从“本地单机版”升级到“双人同步版”时，不需要重写所有页面。

Compose Screen

显示页面，接收用户点击，例如新增愿望、完成日程、发送状态灯。

ViewModel

保存页面状态，处理按钮事件，把业务动作交给 UseCase 或 Repository。

Repository

统一管理数据读写。页面不需要知道数据来自本地还是云端。

Room / DataStore

保存本地业务数据和用户偏好，支撑 V0.1 私密单机原型。

Remote Sync

后期接 Firebase 或 Supabase，实现双人账号、照片备份和共享同步。

数据库设计

妍叶之庭的数据核心不是“页面”，而是事件、回忆和关系联动。Room 数据表要围绕可复用对象设计。

图片不要直接塞进数据库。图片文件放本地私有目录，Room 只保存路径；后期云同步再保存 remoteUrl。

数据表 存什么 典型联动
anniversaries 纪念日、生日、第一次旅行、提醒规则 关联愿望、礼物计划、庆祝回忆
wishes 愿望标题、分类、预算、地点、可见性 愿望转日程，纪念日前推荐礼物
schedules 共同事件、时间、地点、预算、完成状态 完成后转回忆，地点同步到地图
memories 标题、日期、照片路径、心情、地点 回忆进入时间线、地图、纪念日档案
restaurants 餐厅、菜系、忌口、价格、踩雷、最近吃过 今天吃什么转盘，城市美食档案
map_places 省份、城市、点亮状态、避坑、评分 旅行日程完成后自动点亮城市
conflict_reviews 触发点、情绪、诉求、复盘结论、和好时间 生成承诺，进入今日面板待确认
promises 承诺内容、负责人、截止时间、验收人、状态 今日面板提醒，对方确认完成
memos 共享备忘、私密备忘、清单、图片、提醒 关联日程，进入首页今日卡片
care_cycles 姨妈周期、疼痛、情绪、忌口、照顾偏好 本人控制共享范围，对方只看到轻提示

双人同步策略

第一版先不要被云同步绊住。妍叶之庭最重要的是把本地产品闭环跑通，再把 Repository 后面的数据源从纯 Room 扩展成 Room + 云端。

同步是第二阶段能力，不是第一阶段必需品。

Firebase

更适合快速上手

可用于登录、Firestore 云数据库、Storage 图片、推送通知。Android 生态成熟，但国内网络、隐私和成本需要提前考虑。

Supabase

更接近传统后端

基于 PostgreSQL，适合结构化数据和后期可控性。Android 体验不如 Firebase 顺滑，但数据库模型更清晰。

推荐阶段

V0.1：Room 本地单机版
V0.2：账号 + 双人绑定 + 云端备份
V0.3：共享权限 + 状态灯实时同步 + 推送通知
V1.0：端到端加密 + 数据导出 + 纪念册

隐私与边界

情侣 App 最容易做错的地方，是把安全感做成监控感。妍叶之庭应该默认保护敏感内容，分享必须由本人主动选择。

技术设计要服务关系边界，而不是让两个人被功能绑架。

默认私密

敏感数据先保护

吵架复盘、私密愿望、礼物计划、时光信箱、姨妈记录默认仅创建者可见。

部分共享

姨妈记录分层展示

本人看到完整记录；对方只看到本人允许共享的轻提示，例如“今天不舒服”“别安排太累”。

不做查岗

位置和状态可拒绝

不做常驻定位，不做强制拍照。状态灯和下班打卡都应该可关闭、可撤回、可解释。

开发路线

对新手来说，最重要的是不要一开始就把所有浪漫想象同时开工。每一版只解决一个清晰目标。

建议 6-8 周做出可用首版，再根据真实使用频率决定下一步。

第 1 阶段：项目骨架

新建 Android Studio 项目，配置 Kotlin、Compose、Material 3、Navigation、Room、DataStore、Hilt。先完成底部导航和空页面。

第 2 阶段：本地核心功能

实现纪念日、愿望清单、情侣日历、今天吃什么、备忘、姨妈记录。全部使用 Room 本地存储。

第 3 阶段：首页聚合

把最近纪念日、今日日程、待完成备忘、状态提醒、姨妈照顾提示聚合到首页今日面板。

第 4 阶段：跨模块联动

完成愿望转日程、日程转回忆、吃什么转餐厅档案、冲突复盘转承诺等关键闭环。

第 5 阶段：双人版本

加入账号、情侣绑定、共享与私密权限、状态灯、下班打卡、云端备份和通知推送。

Android Studio 起步设置

新建项目时尽量选择现代模板，后续少走弯路。

项目名可以用 YanYeHome，包名建议用 com.yanye.home。

新建项目

推荐选择

  • Template：Empty Activity
  • Language：Kotlin
  • Minimum SDK：API 26 或 API 28
  • Build configuration：Kotlin DSL
  • UI：Jetpack Compose
第一批页面

先做能每天打开的功能

  • HomeScreen：今日面板
  • AnniversaryScreen：纪念日
  • WishScreen：愿望清单
  • ScheduleScreen：情侣日历
  • FoodDecisionScreen：今天吃什么
  • CareScreen：备忘与姨妈记录

一句话原则

妍叶之庭的第一目标不是功能数量，而是让两个人在忙、累、纠结、吵完架的时候，仍然更容易回到彼此身边。技术选型也应该服务这个目标：稳定、私密、可扩展、少折腾。
